## TL;DR — what actually matters

Your app-layer security is genuinely good: the x402 gate (EIP-3009 verify → reserve → serve-once claim), fail-closed admin allowlist, injection-safe resolver, and immutable `events` are all sound. The exposure is **not** in the app — it's in the database's own front door. Supabase ships the `anon` key to the browser (`NEXT_PUBLIC_SUPABASE_ANON_KEY`), so the entire security model rests on Row-Level Security. RLS was correctly blanket-enabled in migration `0004` — and then **two later migrations quietly re-opened it**:

1. **The paid product table is world-readable.** `0006` grants public SELECT on `profile_scenario_responses` — the exact table sold as the Behavioral Response Library. Anyone with the public anon key can dump it directly and never pay. (F1)
2. **The `reviews` table has no RLS at all.** `0010` forgot to enable it, so the anon key can likely read raw signatures and forge reviews directly, bypassing your signature+payment gate. (F2)
3. **Curation can be driven by anyone.** `decide_curation` is `SECURITY DEFINER` and, by Postgres default, executable by `anon` over PostgREST RPC — so an attacker can approve or overwrite pending profiles that then become sellable data. (F3)

On the compounding side, the single biggest issue is that **the Generator OS only actually supports profiles** — the seeded `scenario-gen`/`response-gen` generators can't flow through the executor, so nothing but profiles compounds through generation → hooks → provenance → curation. And **"reproducible by seed," which you advertise on the products, is not implemented** — there is no seed column and no seed is passed to any model.

Fix priority: **F1, F2, F3 before any public launch** (they defeat the paywall and data integrity). Then C1 and C2 (compounding), because those are the whole point of "so next models make useful outputs and everything compounds."

## Security findings

### F1 — The paid table is world-readable (paywall bypass) — CRITICAL

`supabase/migrations/0006_phase2_scenarios.sql`:

```sql
CREATE POLICY "public read profile_scenario_responses"
  ON profile_scenario_responses FOR SELECT TO public USING (true);
```

`profile_scenario_responses` is the table behind **Product #3, the Behavioral Response Library** ($0.03/query, $49 for a 5,000-row pack — `0008`, `0009`). `TO public USING (true)` grants unconditional SELECT to the `public` role, which includes `anon`. Your anon key is intentionally shipped to the browser via `NEXT_PUBLIC_SUPABASE_ANON_KEY` (`src/modules/core/supabase-browser.ts`), so it's readable by anyone who opens the site.

**Impact:** any visitor can call `GET https://<project>.supabase.co/rest/v1/profile_scenario_responses?select=*` with the public anon key and export the entire paid dataset for free, bypassing `src/proxy.ts` completely. The `scenarios` table (the prompts) is exposed the same way; less commercially sensitive but the responses are the product. The `scenario_bias_applications` and `emotional_patterns` exposures are low-value but should be decided deliberately, not by accident.

**Why it happened:** `0004` blanket-enabled RLS on all then-existing tables with **no policies** (correct — deny-all to anon, service role bypasses). `0006` then added `TO public` read policies to make the Lab's browser reads convenient, not realizing the same policy opens the paid PostgREST endpoint to the world.

**Remediation** — drop the public policies; serve this table only through the paid route (which uses the service-role `dbAdmin` and bypasses RLS). If the Lab browser UI needs to read scenarios, gate it behind an authenticated admin policy, not `public`:

```sql
-- 0013_rls_lockdown.sql (excerpt)
DROP POLICY IF EXISTS "public read profile_scenario_responses" ON profile_scenario_responses;
DROP POLICY IF EXISTS "public read scenarios"                  ON scenarios;
DROP POLICY IF EXISTS "public read scenario_bias_applications" ON scenario_bias_applications;
DROP POLICY IF EXISTS "public read emotional_patterns"         ON emotional_patterns;

-- These "service role all" policies are redundant (service_role bypasses RLS)
-- and their current_user check is misleading — remove them too.
DROP POLICY IF EXISTS "service role all profile_scenario_responses" ON profile_scenario_responses;
DROP POLICY IF EXISTS "service role all scenarios"                  ON scenarios;
DROP POLICY IF EXISTS "service role all scenario_bias_applications" ON scenario_bias_applications;
DROP POLICY IF EXISTS "service role all emotional_patterns"         ON emotional_patterns;
-- RLS stays ENABLED with zero policies → anon denied, service_role (the app) unaffected.
```

If you want a genuinely free public *sample* of scenarios for marketing, do it through the existing `/api/v1/preview/[slug]` route (which already returns a deterministic 5% slice), not through a blanket table grant.

### F2 — `reviews` has RLS disabled — CRITICAL

`supabase/migrations/0010_reviews.sql` creates the table and an index but **never runs `ENABLE ROW LEVEL SECURITY`**. The blanket enable in `0004` ran earlier and can't cover a table created six migrations later.

In a default Supabase project the `anon` and `authenticated` roles hold table privileges in the `public` schema, gated only by RLS. With RLS **off**, those grants apply directly, so the anon key can very likely:

- **Read** every row including the raw `signature` column, and
- **INSERT / UPDATE / DELETE** rows directly — bypassing the signature + settled-payment verification your `POST /api/v1/reviews` route so carefully enforces.

That makes review fraud trivial (fabricate 5-star reviews for any `product_slug` with no payment), which is corrosive for a trust-infrastructure product specifically.

**Remediation:**

```sql
ALTER TABLE reviews ENABLE ROW LEVEL SECURITY;
-- No policies → anon denied entirely. All legitimate access already goes
-- through dbAdmin (service role) in the reviews route, which bypasses RLS.
-- If the public site must show reviews client-side, add ONLY:
-- CREATE POLICY "public read reviews" ON reviews FOR SELECT TO public
--   USING (true);   -- but prefer serving them through an API route so the
--                   -- signature column is never exposed.
```

Prefer serving reviews through a server route that selects only `rating, comment, buyer_wallet, created_at` (never `signature`), which the current GET handler already does — so you can keep RLS fully closed.

### F3 — `SECURITY DEFINER` RPCs are anon-callable — HIGH

`0003` defines `decide_curation(...)` as `SECURITY DEFINER` and also `increment_run_counter(...)` and `similar_profile(...)`. In PostgreSQL, **new functions grant `EXECUTE` to `PUBLIC` by default**, and Supabase exposes `public`-schema functions as PostgREST RPC callable by `anon`. Nothing revokes those grants.

`decide_curation` bypasses RLS by design (definer rights) and mutates curation state. An attacker calling `POST /rest/v1/rpc/decide_curation` with the public anon key can:

- **Approve** arbitrary pending profiles (`decision='approved'`) so unreviewed data becomes sellable, or
- **Overwrite** a pending profile's `content`/`big_five`/`summary`/`tags` via `decision='edited_approved'` (it writes straight into `profiles` and stamps provenance as `human-edit`), injecting attacker-chosen content into your paid catalog.

Your admin gate in `src/app/api/lab/decisions/route.ts` (`requireAdmin`) is bypassed entirely because the function is reachable without going through that route. `increment_run_counter` lets anyone corrupt run counters; `similar_profile` leaks approved summaries (lower value, but still).

**Remediation** — revoke execute from the front-facing roles; the app calls these via the service role, which is unaffected:

```sql
REVOKE EXECUTE ON FUNCTION decide_curation(text,uuid,text,text,jsonb,char,text,int,numeric,text,text) FROM PUBLIC, anon, authenticated;
REVOKE EXECUTE ON FUNCTION increment_run_counter(uuid,text) FROM PUBLIC, anon, authenticated;
REVOKE EXECUTE ON FUNCTION similar_profile(text,numeric)   FROM PUBLIC, anon, authenticated;
GRANT  EXECUTE ON FUNCTION decide_curation(text,uuid,text,text,jsonb,char,text,int,numeric,text,text) TO service_role;
GRANT  EXECUTE ON FUNCTION increment_run_counter(uuid,text) TO service_role;
GRANT  EXECUTE ON FUNCTION similar_profile(text,numeric)   TO service_role;
```

(Confirm the exact argument type signatures against your deployed functions — `\df+ decide_curation` in `psql` — before running, since `REVOKE` matches on signature.)

### F4 — `/api/v1/simulate` is an unmetered paid-LLM endpoint — HIGH

`src/app/api/v1/simulate/route.ts` accepts a POST with an arbitrary `scenario` string, injects it into a system prompt, and calls a real LLM (`generateObject`) when provider keys are set. This route is:

- **Not paywalled** — the proxy matcher only covers `/api/v1/query/:path*` and `/lab`.
- **Not rate-limited** — unlike `/api/v1/preview`, it never calls `rateLimit()`.

So anyone can loop POSTs and burn your OpenRouter/Anthropic credits directly (financial DoS), and inject arbitrary text into the model prompt (low stakes here — it only returns demo text — but still uncontrolled). It's the public Playground demo, so it should stay free, but it must be metered.

**Remediation:** add the same IP rate-limit the preview route uses, cap `scenario` length, and consider a tighter per-IP budget for LLM-backed calls:

```ts
import { rateLimit, clientIp } from '@/modules/core/rate_limiter';
// ...
if (!(await rateLimit(`sim:${clientIp(req)}`))) {
  throw err('too_many_requests', 429, 'Rate limit exceeded.');
}
if (typeof scenario !== 'string' || scenario.length > 1000) {
  throw err('invalid_params', 400, 'scenario must be a string under 1000 chars');
}
```

Consider a dedicated lower limit (e.g. 10/min) for LLM-backed routes rather than sharing the 60/min bucket.

### F5 — New tables aren't covered by the blanket RLS enable — MEDIUM (process)

The `DO $$ ... ENABLE ROW LEVEL SECURITY ... $$` loop in `0004` only affected tables that existed at that moment. Every table created afterward must enable RLS itself. That contract has **already been violated twice** — `0006` (had to enable manually, and did, but then over-granted) and `0010` (forgot entirely, → F2). This will keep happening, including with the two new product tables in this same session, unless it's made systematic.

**Remediation (pick one, ideally both):**

1. A migration convention: every new migration that runs `CREATE TABLE` ends with `ALTER TABLE <t> ENABLE ROW LEVEL SECURITY;` and an explicit policy decision (even if "no policy = deny all").
2. A CI guard that fails the build if any `public` table has `rowsecurity = false` (query in the Appendix), so a forgotten enable can't merge. This is the durable fix — it turns F2-class mistakes into red CI instead of silent exposure.

## Compounding findings — "so next models make useful outputs and everything compounds"

### C1 — The Generator OS only actually supports profiles — HIGH

`src/modules/generation/executor.ts` → `processRun()` unconditionally calls `insertProfile(item, ...)` and inserts provenance with `entity_type: 'profile'`, regardless of the generator's `entity_type`. The seeded `scenario-gen` (entity `scenario`) and `response-gen` (entity `profile_scenario_response`) generators exist as rows (`0006`) but **cannot run through this executor** — their output would be shoved into the `profiles` table and fail or corrupt.

Consequence: only profiles compound through the intended loop (generate → hooks → provenance → curate → freeze → sell). Scenarios and responses must be inserted by some out-of-band path, which means they lack run-linked provenance and never pass the hook chain. This directly caps the thing you asked to protect: as you generate more data, only one entity type actually benefits from the pipeline's quality and traceability guarantees.

**Remediation:** dispatch the insert + provenance step on `gen.entity_type`, mirroring how `resolveQuery` and the preview route already switch on `recipe.entity`. Sketch:

```ts
// in processRun, replace the hardcoded insertProfile + provenance block:
const entityId = await insertEntity(gen.entity_type, item, verdict, run.id);
await stampProvenance(gen, run, prompt, item, entityId, input.params);

// insertEntity switches: 'profile' -> insertProfile (existing),
// 'scenario' -> insert into scenarios, 'profile_scenario_response' ->
// insert into profile_scenario_responses (resolving profile_id/scenario_id
// from params), etc. Each returns the new id; provenance is stamped uniformly.
```

This is the highest-leverage compounding fix: once the pipeline is entity-generic, every new product type you add (including the Robinhood scenario batteries) inherits hooks, provenance, curation, and dataset-freezing for free — which is exactly the MASTERPLAN's promise.

### C2 — "Reproducible by seed" is advertised but not implemented — MEDIUM

Multiple product descriptions say "reproducible by seed" (e.g. `0008` Behavioral Response Library; the MASTERPLAN's design principle #6 is "same seed + same versions = byte-identical output"). In reality:

- `generation_runs` (`0001`) has **no `seed` column** (the MASTERPLAN's draft DDL had one; the shipped table dropped it).
- `generateItems()` (`src/modules/generation/models.ts`) passes **no seed** to any provider; it only sets `temperature`.

So identical inputs do not produce identical outputs, and nothing records a seed to reproduce from. This contradicts design principle #7 ("everything is honest") and undermines the premium Benchmark & Evaluation product (Product #7) whose entire pitch is seeded reproducibility.

**Remediation, two honest options:**

1. **Implement it:** add `seed INT` to `generation_runs` and to `provenance.params`, pass `seed` to providers that support it (OpenAI/OpenRouter `seed`, Google not deterministic), and record it. Note that even with a fixed seed, most hosted LLMs are only *best-effort* deterministic — so pair seeds with frozen `datasets` (content-hashed membership, which you already have) as the real reproducibility guarantee, and reproduce by *replaying the dataset*, not by re-sampling the model.
2. **Or soften the copy** to what's actually true today: "provenance-stamped and content-hashed; frozen datasets are byte-stable." Don't claim seed-determinism until the plumbing and a determinism test exist.

Given the Robinhood certification thesis leans hard on reproducibility, option 1 (seed column + frozen-dataset replay) is worth doing before you market on it.

### C3 — Recipe hard-filters are ignored → `solana-trading-pack` serves the wrong data — MEDIUM

`resolveProfileQuery` (`src/modules/recipes/resolver.ts`) only reads `rules.filters?.status`; it never applies a recipe-level `filters` array. But `0012` defines the Solana pack recipe as:

```json
{ "entity": "profile", "filters": [ { "field": "tags", "operator": "contains", "value": ["chain:solana"] } ] }
```

That filter is silently dropped. A buyer paying $0.05–$15 for the "Solana Trading Psychology Pack" receives **all approved profiles**, not Solana-tagged ones — and conversely the general library and the Solana pack return identical data. It's both a commercial-integrity bug (paying for a curated slice you don't get) and a latent correctness trap for every future themed pack, including the Robinhood counterparty pack you're about to add.

**Remediation:** honor a server-side, non-overridable hard filter in the resolver, e.g. `rules.filters.tags_include` applied with `.overlaps('tags', ...)` before request filters. (I'll build the new Robinhood pack on this corrected path so it's right from day one.)

### C4 — Dedup is profile-only and misses paraphrases — MEDIUM

`dedup` (`src/modules/hooks/dedup.ts`) calls `similar_profile`, which uses pg_trgm `similarity()` on `summary` at threshold **0.55** (seed) / 0.55 default. Trigram similarity rarely exceeds ~0.3–0.4 for semantically-equivalent but differently-worded text, so near-duplicates with fresh wording pass straight through. It also only exists for profiles. As volume grows, silent duplicate accumulation dilutes dataset value and skews the population statistics you sell on.

**Remediation:** either lower/calibrate the trigram threshold against a labeled dup set, or move to embedding-based similarity (store an embedding per entity, dedup on cosine distance). At minimum, add a dedup path for each new entity type so responses/scenarios don't accumulate dups.

### C5 — No `distribution_check` hook; `route` scoring is coarse — LOW

The MASTERPLAN treats distribution validation against IPIP-NEO norms as a pillar, but there's no `distribution_check` hook in the registry. And `route` averages hook `score`s where `schema_validate`/`provenance_stamp` contribute none and `dedup` contributes 1/0, so `auto_approve_above` operates on a noisy near-binary mean. Today `auto_approve_above` is `null` (everything queues for human review), so this is latent — but the moment you enable auto-approve to scale, quality control is thinner than it looks.

**Remediation:** implement a `distribution_check` hook (batch-level KS/z-test of generated Big Five means/sds vs published norms) and give `route` an explicit, documented scoring policy (e.g. weighted, with judge score dominating) before enabling auto-approve.

### C6 — `provenance_stamp` hook is a no-op — LOW (clarity)

`src/modules/hooks/provenance.ts` returns `{ passed: true }` and does nothing; the real provenance insert lives in the executor. Functionally fine for profiles, but misleading: a new generator that lists `provenance_stamp` in its hook chain gets no provenance from the hook itself. Fold provenance stamping into the entity-generic insert path (C1) and either make the hook real or remove it from chains to avoid implying a guarantee it doesn't provide.

## What's already solid (so it's not touched)

- **x402 gate** (`src/proxy.ts` + `payment-verify` + `claimServe`): EIP-3009 off-chain verify, unique-`payment_sig` replay guard (`0005`), reserve-before-serve, tier-bound single-use claim, fail-closed on infra errors, production-forced payment binding. This is careful, correct work.
- **Admin auth** (`requireAdmin`): fail-closed on an empty `ADMIN_EMAILS` allowlist.
- **Injection safety**: the resolver whitelists request filters (`allow_request_filters`) and only curators author prompts; buyer input never reaches an LLM prompt on the paid path.
- **Immutability**: `events` has `DO INSTEAD NOTHING` rules on UPDATE/DELETE — a real append-only audit log.
- **Curation atomicity**: `decide_curation` uses `FOR UPDATE` + versioning + provenance on edits. (Its only problem is *who can call it* — F3, not *what it does*.)

## Appendix — verification probes

Run in the Supabase SQL editor (as a low-privilege check, these read catalog state):

```sql
-- A) Any public table with RLS OFF (F2/F5 detector — should return zero rows):
SELECT relname
FROM pg_class c JOIN pg_namespace n ON n.oid = c.relnamespace
WHERE n.nspname = 'public' AND c.relkind = 'r' AND c.relrowsecurity = false;

-- B) Any policy granting SELECT to anon/public (F1 detector — review each):
SELECT tablename, policyname, roles, cmd, qual
FROM pg_policies
WHERE schemaname = 'public' AND 'public' = ANY(roles) OR 'anon' = ANY(roles);

-- C) Functions executable by anon/public (F3 detector):
SELECT p.proname, pg_get_function_identity_arguments(p.oid) AS args,
       has_function_privilege('anon', p.oid, 'EXECUTE') AS anon_can_execute
FROM pg_proc p JOIN pg_namespace n ON n.oid = p.pronamespace
WHERE n.nspname = 'public'
  AND p.proname IN ('decide_curation','increment_run_counter','similar_profile');
```
